# Deployment Patterns

## Release Pipeline
- Full pipeline: `gh workflow run ci-release.yml` → wait for RC → promote locally → `gh workflow run ci-deploy.yml -f version=X.X.X`. Never skip promote. Never run `sodium deploy` locally.
- RCs are kept as pre-releases (not deleted). Promote copies release notes to final release.
- Version bumps happen automatically during `sodium release`. Never manually bump `package.json` versions.

## Image Building & Content Hash
- Release uses hash-based RETAG/SKIP for backend + infra-worker: if the exact tag exists in GHCR → SKIP; if the same content hash exists under an older version tag → pull + retag + push (no rebuild). Retagged images are smoke-tested too. To force a fresh build use `sodium release --force-build` (also forces the frontend build).
- Content hash covers the app dir + `@sodium/*` workspace deps PLUS out-of-tree build inputs: `pnpm-lock.yaml` (all images), `docker/base/Dockerfile.base` + `scripts/ci/package-for-docker.sh` (backend). In-app Dockerfiles (`apps/*/docker/`) are covered by the directory walk. Version bumps do NOT change the hash (package.json `version` field is excluded).
- Retagging is version-safe: the backend reports the deploy-time `VERSION` env var (set from the release manifest by Pulumi in `cloudrun-adapter.ts`) in `/readyz`, `/health` and appConfig — the package.json baked into a retagged image is only a fallback (`appVersion` in `apps/backend/src/consts.ts`).
- GHCR layers can have corrupted checksums. If `docker push` fails with `file integrity checksum failed`, bump version and re-run CI.
- Use plain `docker build`, NOT `docker buildx build` (remote buildx driver can't see local images).

## CI Environment
- Backend typecheck needs `NODE_OPTIONS='--max-old-space-size=4096'` (CI default 2GB heap OOMs).
- Turbo `typecheck` task must have `dependsOn: ["^build"]` so workspace deps are built first.
- CI workflows that push tags must disable git hooks: `git config core.hooksPath /dev/null`.
- Use `silent: !process.env.CI` — `silent: true` swallows all error output in CI.
- CI deploy workflow must build all workspace deps needed by infra-worker: `pnpm turbo build --filter=sodium-cli --filter=@sodium/infra-worker`.

## Docker Packaging
- `package-for-docker.sh` must include ALL runtime-needed files (dist/, prisma/, pgroll-migrations/, etc.).
- Release manifests are env-agnostic — GCP URLs resolve at deploy time from `--env` flag.
- Frontend artifacts use versioned filenames: `frontend-build-{version}.tar.gz`.

## Deploy Safety
- **NEVER run `stack-runner.ts` directly** or raw `pulumi up` — causes Cloud Run service deletion.
- **ALWAYS deploy via CI:** `gh workflow run ci-deploy.yml -f version=X`. No DEV environment — deploy directly to prod.
- CI deploy SA needs Owner role on BOTH GCP projects (`sodium-shared-platform` and `goatlab-sodium-state-prod`) — legacy key-based auth path only.
- **GCP auth in CI is a soft switch (WIF):** the three GCP workflows (`ci-deploy`, `ci-promote`, `release-rollback`) authenticate via the composite action `.github/actions/gcp-auth` — Workload Identity Federation (keyless, repo-scoped `ci-deployer` SA with a concrete role list instead of Owner) when repo variables `WIF_PROVIDER`/`WIF_SERVICE_ACCOUNT` are set; legacy `GCP_SA_KEY`/`GCP_CREDENTIALS` JSON key when unset. WIF needs `permissions: id-token: write` on the job. Pool/provider/SA live in the bootstrap stack (`apps/infra_new/src/stacks/bootstrap/github-wif.ts`) and are created **unconditionally** — the bootstrap stack redeploys on every ci-deploy, so a flag-gated block would be DELETED by the next routine deploy. Activation/rollback = repo-variable flip: `apps/infra_new/docs/wif-activation.md`. The GitHub pool is separate from the k3s ESO pool (`k3s-goat` — cluster OIDC issuer via uploaded JWKS); never merge them.

## Pipeline Behavior (since 2026-06)
- **Continuous deployment from main:** any push to `main` with deployable changes auto-runs `ci-release.yml` → on green, finalize dispatches `ci-promote.yml` (`auto_deploy=true`) → promote dispatches `ci-deploy.yml` to prod. Disable with repo variable `AUTO_DEPLOY=off` (`gh variable set AUTO_DEPLOY -b off`); absent variable = enabled. The `paths-ignore` list (docs, `.planning/`, `.claude/`, `release_notes/`, `infra/`, `cluster/`, legacy/separately-deployed apps) lives in the `push:` trigger of `.github/workflows/ci-release.yml`. Loop safety: the pipeline's own commits to main (version bumps, Pulumi state) carry `[skip ci]`, which GitHub honors for push triggers.
- Deploy/rollback workflows share concurrency group `deploy-{env}` — concurrent prod deploys queue, never overlap (protects Pulumi state commits to main).
- **Bootstrap state backend opt-in (prepared 2026-06):** bootstrap stacks (`sodium-bootstrap-prod`, `sodium-platform-oauth`) use the git-committed file backend `apps/infra_new/pulumi-state-production/` UNLESS the committed marker `pulumi-state-production/backend-migrated.json` exists — then every resolver/reader (bootstrap-backend-resolver, secrets-retriever/manager, bootstrap-runner, sodium-cli `util/bootstrapState.ts`, backend `proxy.ts`) targets the GCS bucket from the marker, and the "Commit Pulumi state" steps in `ci-deploy.yml`/`release-rollback.yml` skip themselves. Marker malformed = loud `MalformedBootstrapMarkerError` (never silent fallback). Migration script: `apps/infra_new/scripts/migrate-bootstrap-state-to-gcs.ts`; runbook + rollback: `apps/infra_new/docs/bootstrap-state-migration.md`.
- Image promotion GHCR → GCP AR uses `crane copy` (no docker pull/push); `Promote Release` pre-copies images so deploy's promotion is an idempotent no-op. crane needs the docker config auth that `docker/login-action` + `gcloud auth configure-docker` set up.
- **No-traffic canary rollout (since 2026-06):** the backend Cloud Run deploy is two-phase Pulumi. Stage: new revision at 0% traffic + `candidate` tag, 100% pinned to the currently-serving revision (read live via `gcp.cloudrun.getService` inside the program — pin to the revision RECEIVING traffic, not `latestReadyRevisionName`, since a rejected candidate can be latest-ready). Gate: poll the candidate's tagged URL (`https://candidate---<host>/readyz`) — failure aborts with prod traffic untouched. Promote: traffic-only `pulumi up` flips 100% → latest. State stays truthful in every phase (no `ignoreChanges`, no gcloud). Bootstrap (fresh service) falls back to direct 100%. Instant rollback: `sodium deploy --env prod --release <current-version> --rollback-traffic <revision>`. Escape hatches: `--no-canary` (single-phase legacy, also for wedged traffic state), `--skip-health-check`.
- Deploy health-gates on `/readyz` (canary candidate URL pre-cutover, then a quick public re-verify post-flip; 3-min timeout, fails the workflow). `--skip-health-check` to bypass. Release deploys also assert the `/readyz` version (warn-only until PR #243's manifest VERSION env injection lands — baked package.json version legitimately diverges on smart-cache retag/reuse deploys).
- **Canary outputs must fail closed (v2026.06.10.2 incident):** the stage→promote handoff rides on stack outputs. Never derive the candidate URL from the provider-reported `traffics[].url` (not reliably populated in resource state) — build it deterministically as `https://<tag>---<host>` from `statuses[0].url`. Never emit canary outputs that can resolve to `undefined` (Pulumi silently DROPS undefined-valued stack outputs) — use `''` sentinels plus an explicit `backendCanaryStaged` boolean. The CLI must treat missing/ambiguous outputs after a stage run as a HARD failure, never as "bootstrap": inferring bootstrap from an absent output made the deploy report success while the staged candidate sat at 0% traffic and prod kept serving the old release. Recovery from a half-staged state needs no manual cleanup — the next deploy's stage phase pins to the revision receiving traffic and replaces the whole traffic list (stale candidate tag included).
- **Post-flip observability gate (2026-06):** the `/readyz` health gate proves the revision boots, not that it survives real traffic. After the canary flip, the deploy watches Cloud Monitoring (`--observability-window <seconds>` / `--observability-warmup <seconds>` / `--skip-observability-gate`): the new revision's 5xx rate and p95 vs the previous revision's trailing 30-min baseline; FAIL when above max(2× baseline, 2% / 2s) with ≥ 20 requests observed (low traffic PASSES — quiet hours must not block deploys; fail-fast mid-window on a conclusive breach). On FAIL it auto-pins traffic back to the previous revision (canary `pin` phase) and strictly verifies the public `/readyz` reports the previous version before failing the workflow. Metric-READ failures (auth/API) only warn — never roll back on missing data. Pure logic + tests: `packages/sodium-cli/src/util/deploy/observability-gate.utils.ts`.
- **Cold starts are not regressions (v2026.06.10.9 false positive):** the gate's first live run rolled back a healthy deploy — the new revision's first 41 requests all hit cold instances (p95 2393ms vs threshold 2151ms at 92s) while the baseline was a 30-min-warm revision. Fixes: (1) warm the candidate via its tagged URL BEFORE the flip (~30 best-effort requests over ~20s — `warm-up.ts`); (2) the gate ignores the first 60s post-flip (`--observability-warmup`; evaluation window starts at flip+60s, total watch = warmup + window); (3) a latency breach must persist across 2 consecutive polls before rolling back (5xx still fails immediately; absent p95 data is never a breach). Any future gate threshold must ask: "does this metric look different on a cold instance?"
- **Baseline-relative thresholds need absolute ceilings (v2026.06.10.10):** `run.googleapis.com/request_latencies` includes long-lived streaming/long-poll/SSE requests — a 30-min baseline window read p95 ~303570ms (~5 min!) for a revision that measured 1076ms an hour earlier, so the 2×-baseline rule produced a 607s threshold and silently disabled the latency check for that deploy. The effective threshold is now `min(max(2× baseline, 2s), 10s ceiling)` (`--observability-p95-ceiling <ms>`), with a loud "baseline looks polluted by long-lived requests" log whenever the clamp engages. p50 as a secondary signal was considered and deliberately rejected: requiring p50 confirmation would blind the gate to genuine tail-only regressions (slow queries/N+1 surface in the tail first). Lesson: any "N× baseline" comparison against a distribution mixing interactive and long-lived requests must be bounded by absolute floors AND ceilings.
- pgroll contract is **lagged one release**: deploy completes the previous release's pending migration before expanding new ones, and leaves its own migration active. Backend rollback stays safe until the next deploy. `contract_now` workflow input / `--contract-now` restores immediate contract. Deploy-time migration validation was removed from `ci-deploy.yml` (it validated main HEAD, not the release — validation runs in `ci-release.yml`).
- Release skips the frontend build when `git diff` vs the previous release tag shows no frontend-relevant changes (`apps/frontend/`, `packages/`, lockfile, workspace file; version-bump-only commits excluded). Previous artifact is re-attached; manifest shows `frontend.reusedFromVersion`. `--force-build` overrides.

## GitHub Actions / CD Chain Gotchas (earned 2026-06-10)
- **Never write the literal bracketed skip-ci marker in PR titles/bodies** — squash merges use the PR body as the commit message and GitHub honors the marker anywhere in it. PR #240 (the PR that ENABLED CD) skipped its own first CD trigger this way. Write "skip-ci marker" in words.
- **GitHub templates `action.yml` description fields too:** literal `${{ vars.* }}`/`${{ secrets.* }}` expressions in a composite action's input *descriptions* fail template validation at runtime (composite actions can't read those contexts). Reference variables in plain text in docs strings; only `${{ inputs.* }}` belongs in composite action expressions.
- **The releases API intermittently 401s the Actions token** (two consecutive cycles; a PAT worked at the same moment, githubstatus green). The "Publish draft RC" step retries 5×/20s. Manual unblock: publish locally (`gh release edit vX-rc --draft=false --prerelease`) then `gh workflow run ci-promote.yml -f rc_tag=vX-rc -f auto_deploy=true`.
- **Publish → promote is eventually consistent:** a promote dispatched immediately after flipping a draft can get "RC not found" from the API. `promoteRelease` retries the lookup 4×/45s (release.ts).
- **Re-running a failed finalize can orphan the draft's assets** (drafts aren't bound to their tag until published) → a published RC with 0 assets that fails promote at "Downloading RC assets". Recovery: demote it back to draft (`gh release edit vX-rc --draft`) and let the next cycle supersede; don't hand-re-attach assets. Full table: deployment.md "CD failure modes & recovery".

## Test Gates in Release (since 2026-06)
- `ci-release.yml` has 3 jobs: `release` (typecheck → unit tests → build → container boot smoke test → GHCR push → DRAFT RC), `full-tests` (functional + integration via testcontainers, parallel), `finalize` (publishes draft only when both green). Draft RC = unpromotable; `release --promote` refuses drafts, lookups use `--exclude-drafts`.
- Smoke test boots the freshly built images BEFORE GHCR push (`packages/sodium-cli/src/util/images/smoke.ts`, `--skip-smoke-test` to bypass). Env gotchas learned the hard way: backend must use `APP_ENV=e2e` (`test` mode never calls `app.listen()` — runScript exits cleanly with "Script completed"); infra-worker ignores the `PORT` env entirely (listens on 4324 unless `K_SERVICE` is set — Cloud Run marker); its Dockerfile HEALTHCHECK targeting `:8080` is always unhealthy outside Cloud Run.
- Backend image smoke needs: `APP_ENV=e2e`, `MOCK_SECRETS=true`, a redis sidecar (`REDIS_URL` + `REDIS_TEST_CONNECTION_STRING`), `TYPESENSE_URL=https://...` + mock keys (base image bakes `NODE_ENV=production` → Typesense client enforces HTTPS at module scope).
- **Built-image HTTP tests (since 2026-06):** after the boot smoke test and before GHCR push, freshly BUILT backend images run real HTTP flows against Postgres(pgvector)+Redis sidecars (`packages/sodium-cli/src/util/images/http-tests.ts`, `--skip-image-tests` to bypass; retags skip automatically). Covers /health, /livez, /readyz?tenant=, public tRPC, 401/404 tRPC error envelopes, and a full Better Auth OTP sign-in → bearer → `user.me`. ~40s locally. Gotchas earned here:
  - **Schema provisioning runs INSIDE the image:** `pgroll init` + `pgroll migrate --complete` with the image's bundled pgroll binary and `/app/pgroll-migrations` (shared/ + tenants/sodium-platform merged into one dir, filename order — same as `migrateLeader()`). Do NOT use host `prisma db push`: in Prisma 7 it regenerates the prisma client with the tenant-composed schema as a side effect, which races the parallel frontend build in the release job.
  - **`sodium-platform` is the ONLY tenant in e2e mode** — `sodium-test` is registered only when `APP_ENV=test` (tenant.registry.ts). In e2e, ALL tenants resolve their DB from `process.env.DB_URL` (checked first by `getDatabaseUrl`) → point it at the sidecar.
  - **`SECRET_DB_URL` must ALSO be set:** `delphi.config.ts` builds its pg pool from the DB_URL *secret*, not the env var. With MOCK_SECRETS the mock provider returns `''` → pg-pool silently dials localhost:5432 and every endpoint that enqueues a delphi workflow (e.g. first-login account creation in `user.me`) 500s.
  - **`system_roles` must be seeded before any login:** account auto-creation FK-fails on `roles.APP_USER.id` otherwise. Role rows are seeded at tenant creation in prod (rolesSeeder), never by migrations — the HTTP test stage inserts the 6 fixed roleIds from shared-schemas via psql.
  - **Better Auth in e2e:** test emails (`test*@test.getsodium.com`) get fixed OTP `000000` and skip email dispatch; the bearer plugin returns the session token in the `set-auth-token` response header. Override `SECRET_BETTER_AUTH_SECRET` (mock provider returns empty).
  - **Dockerfile.prebuilt arch portability:** pgroll download uses `TARGETARCH` (was hardcoded amd64 — broken on Apple Silicon local builds), and `COPY --link --chown` must use numeric `1001:1001` (named users can fail to resolve on newer BuildKit with "invalid user index: -1" since --link layers build independently of the base).
